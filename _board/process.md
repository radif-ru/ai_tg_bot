# Процесс выполнения одной задачи

Документ описывает пошаговый алгоритм для закрытия **одной** задачи из активного спринта. Алгоритм универсален — не зависит от конкретной задачи. Содержание задачи, её Definition of Done и связанные документы берутся из `_board/sprints/<active>.md` и `_docs/`.

## 0. Перед началом сессии

Один раз на сессию работы:

1. `git status` пуст, ветка актуальна (`git pull`).
2. Активировано виртуальное окружение: `source .venv/bin/activate`.
3. Зависимости актуальны: `pip install -r requirements.txt` (если был добавлен/обновлён модуль).
4. Ollama запущена: `ollama list` отвечает.
5. На активной ветке спринта (см. `_board/plan.md` § «Индекс спринтов» → колонка «Ветка»).

## 1. Прочитать документацию

**Обязательный шаг перед началом любой задачи.**

1. `CLAUDE.md` — поведенческие гайдлайны LLM-агента (think before coding, simplicity, surgical changes, goal-driven). Применяются к любой задаче.
2. `_docs/README.md` — навигация по проектной документации.
3. `_docs/current-state.md` — есть ли уже запись про затрагиваемую область (потенциальные подводные камни, легаси).
4. `_docs/instructions.md` — правила разработки (git, стиль, async, ошибки, секреты, тесты). Перечитывать до автоматизма.
5. `_board/plan.md` — индекс спринтов: какой спринт активен и какая у него ветка.
6. `_board/sprints/<active>.md` — выбираемая задача и её соседи (зависимости).
7. Документы из поля «Связанные документы» выбранной задачи (например, `_docs/architecture.md` §3.4 для задач по LLM-клиенту).

При расхождении документа и кода — приоритет у кода. Документ обновится после задачи отдельным коммитом.

## 2. Выбрать задачу

1. Открыть `_board/plan.md` → найти активный спринт → открыть его файл `_board/sprints/<NN>-<short-name>.md`.
2. Найти **первую сверху** задачу со статусом `ToDo` (по этапам: сначала задачи этапа 1, затем этапа 2, …).
3. Проверить поле «Зависит от»: все указанные задачи должны быть `Done`. Если нет — взять предыдущую (по зависимостям, не по порядку).
4. Если требуются внешние действия (ТЗ, доступы, решения от пользователя), которых нет — пометить задачу `Blocked` с причиной, остановиться.

## 3. Перевести задачу в статус `Progress`

1. В файле спринта `_board/sprints/<NN>-<short-name>.md` у выбранной задачи изменить `**Статус:** ToDo` → `**Статус:** Progress`.
2. Обновить «Сводную таблицу задач спринта» в этом же файле (колонка «Статус»).
3. Закоммитить:
   ```bash
   git add _board/sprints/<NN>-<short-name>.md
   git commit -m "chore(plan): start task <NN>.<stage>.<task>"
   # пример: chore(plan): start task 02.1.4
   ```

> Правило: в статусе `Progress` одновременно — **ровно одна** задача на всей доске.

## 4. Подготовить контекст для реализации

1. Прочитать «Описание» и «Definition of Done» задачи целиком.
2. Открыть в редакторе файлы из секции «Затрагиваемые файлы» (если указана) или предполагаемые по описанию.
3. Если задача связана с конкретной подсистемой — открыть параллельно соответствующий документ из `_docs/` (`architecture.md` / `commands.md` / `stack.md` / `testing.md`).
4. Сформулировать (для себя или для LLM-помощника) краткий план шагов.

## 5. Реализовать задачу

Следуем `_docs/instructions.md` (минимализм, хирургические правки, цель → проверка):

1. **Самый минимальный диф**, решающий проблему.
2. **Не править соседний код** без явной задачи на это.
3. **Не добавлять «гибкости на будущее»** — реализуй ровно то, что нужно.
4. **Соблюдай существующий стиль** (импорты, нейминг, структура файлов).
5. Если встретился несвязанный баг или мёртвый код — записать в `_docs/current-state.md` §2 по шаблону из §5, **не править** в этой задаче.

Цикл «правка → проверка»:

1. Внести правку.
2. Прогнать smoke-проверки (см. §6).
3. Если что-то поломалось — откатить и переделать.
4. Если предусмотрены тесты — добавлять/обновлять до зелёного состояния (см. §6.3).

## 6. Проверка изменений

### 6.1 Минимальный smoke-test (всегда)

```bash
# 1) Импорт пакета не сломан
python -c "import app; print('OK')"

# 2) Сборка приложения (точка входа) — если затронуты handlers / main / services
python -c "import asyncio; from app.main import main; print(main)"

# 3) Один сценарий вручную (для задач, меняющих UX) — в реальном Telegram:
python -m app
# отправить /start и одно текстовое сообщение
# Ctrl+C
```

### 6.2 Целевые проверки (когда применимо)

- **Конфигурация** — `python -c "from app.config import Settings; print(Settings().ollama_default_model)"`.
- **LLM-клиент** изменён — `pytest tests/services/test_llm_client.py -q`.
- **Handlers** изменены — `pytest tests/handlers/ -q` + ручная проверка соответствующей команды в Telegram.
- **Middleware/Errors** — намеренно падающий хендлер локально → бот должен ответить «Что-то пошло не так», polling не падает.
- **`.env` / секреты** — `git status` не показывает `.env`; `git log -p | grep TELEGRAM_BOT_TOKEN=` возвращает только примеры из `.env.example`.

### 6.3 Тесты

```bash
pytest -q                                      # все тесты
pytest tests/<затронутый-модуль> -q -v         # детально по подсистеме
pytest --cov=app --cov-report=term-missing     # покрытие (если установлен pytest-cov)
```

## 7. Проверить Definition of Done

1. Открыть в файле спринта раздел «Definition of Done» выбранной задачи.
2. По каждому пункту — отметить «выполнено / не выполнено».
3. Если хотя бы один пункт не закрыт — вернуться к §5.
4. Если в DoD есть «обновить документацию X» — обновить и проверить, что `_docs/README.md` § «Навигация» и связанные ссылки всё ещё корректны.
5. Прогнать `git status` — никаких лишних файлов (`*.log`, `.env`, `__pycache__/`).

## 8. Зафиксировать результат в git

Коммиты — атомарные, conventional commits, в императиве:

```bash
git add <изменённые файлы>
git commit -m "feat(<scope>): <что сделано>"
# или: fix(...) / refactor(...) / docs(...) / test(...) / chore(...)
```

Примеры (типичные scope этого проекта):

- `feat(handlers): add /reset command`
- `fix(llm): map asyncio.TimeoutError to LLMTimeout`
- `refactor(messages): extract long-message splitter call`
- `docs(architecture): clarify error handler flow`
- `test(registry): cover reset_prompt isolation`
- `chore(plan): complete task 02.1.4`

## 9. Перевести задачу в статус `Done`

1. В файле спринта изменить `**Статус:** Progress` → `**Статус:** Done`.
2. Обновить «Сводную таблицу задач спринта».
3. Дописать строку в § «История изменений спринта»: дата + краткое описание + ссылка на коммит/коммиты.
4. Если это последняя задача спринта — обновить также `_board/plan.md` § «Индекс спринтов» (перенести спринт из «Активный» в «Закрытые», заполнить дату закрытия) и сводную таблицу в `_board/plan.md`.
5. Коммит:
   ```bash
   git add _board/sprints/<NN>-<short-name>.md _board/plan.md
   git commit -m "chore(plan): complete task <NN>.<stage>.<task>"
   ```
6. Пуш (или открыть PR, если работали в ветке):
   ```bash
   git push -u origin <branch>
   ```

## 10. Обновить документацию (если нужно)

Если в ходе задачи изменилось поведение, отражённое в `_docs/`:

1. Найти соответствующий документ (`commands.md`, `architecture.md`, `stack.md`, `testing.md`).
2. Внести правку.
3. Если изменилась структура каталога или появился новый документ — обновить `_docs/README.md` § «Навигация» и `_docs/project-structure.md`.
4. Если найден новый легаси-нюанс — добавить в `_docs/current-state.md` §2 (см. шаблон в §5 этого документа).
5. Коммит `docs(<scope>): ...`.

## 11. Перейти к следующей задаче

1. Вернуться к §2 — выбрать следующую задачу `ToDo`.
2. Если все задачи спринта `Done`, а пунктов ToDo нет — спринт можно закрывать (см. `_board/plan.md` § «Правила работы со спринтами», п.4–5).
3. Если есть запрос пользователя на новый спринт — открыть новый `sprints/<NN+1>-<short-name>.md` (новая ветка `feature/<short-name>` от `main`).

## Короткая шпаргалка (TL;DR)

1. Прочитать `CLAUDE.md` → `_docs/README.md` → `_docs/current-state.md` → `_board/plan.md` → файл активного спринта → документы задачи.
2. Взять верхнюю `ToDo` задачу — проверить, что зависимости `Done`.
3. Сменить статус на `Progress` в файле спринта, коммит `chore(plan): start task <NN>.<stage>.<task>`.
4. Минимальный диф → smoke-test → зелёный.
5. По каждому пункту DoD — отметка «выполнено».
6. Коммит правок (`feat/fix/refactor/docs/test/chore`).
7. Сменить статус на `Done`, коммит `chore(plan): complete task <NN>.<stage>.<task>`, пуш.
8. К следующей задаче.

## Чек-лист перед переходом к следующей задаче

- [ ] Definition of Done закрыт по всем пунктам.
- [ ] `pytest -q` зелёный.
- [ ] Smoke-test пройден (импорт + ручная проверка в Telegram, если задача меняет UX).
- [ ] `git status` чист (нет лишних артефактов / `.env` / логов).
- [ ] Документация обновлена там, где меняется поведение.
- [ ] Статус задачи в файле спринта = `Done`, сводная таблица актуальна.
- [ ] Если последняя задача спринта — `_board/plan.md` обновлён (спринт переведён в Closed).
- [ ] Коммиты запушены.
