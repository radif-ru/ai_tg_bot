# Процесс выполнения одной задачи

Документ описывает пошаговый алгоритм разработчика (и/или LLM-ассистента) для закрытия **одной** задачи из `_board/plan.md`. Алгоритм универсален — не зависит от конкретной задачи. Содержание каждой задачи, её Definition of Done и список затрагиваемых файлов берутся из `_board/plan.md` и `_docs/`.

## 0. Перед началом работы

Один раз на сессию:

1. Убедиться, что репозиторий в чистом состоянии: `git status` пуст, ветка актуальна (`git pull`).
2. Активировать окружение: `source .venv/bin/activate` (если `.venv` уже создан).
3. Убедиться, что запущен Ollama: `ollama list` отрабатывает.

## 1. Прочитать документацию

**Обязательный шаг перед началом любой задачи.**

1. Прочитать `CLAUDE.md` в корне репозитория — поведенческие гайдлайны LLM-агента (think before coding, simplicity, surgical changes, goal-driven). Применяются к любой задаче.
2. Открыть и прочитать `_docs/README.md` — сориентироваться в структуре документации.
3. Прочитать `_board/plan.md` целиком — понять общий контекст и зависимости.
4. Прочитать `_docs/instructions.md` — правила разработки (git, стиль, async, логирование, секреты, тесты). Эти правила применяются к **любой** задаче.
5. Открыть `_docs/` и прочитать документы, указанные в разделе «Связанные документы» выбранной задачи.
6. Если что-то в `_docs/` противоречит `_docs/RAW.md` — приоритет у `RAW.md` (`_docs/README.md` §«Источник истины»).

## 2. Выбрать задачу

1. Открыть `_board/plan.md`, найти **первую сверху** задачу со статусом `ToDo`.
2. Проверить, что все задачи, указанные в её поле «Зависит от», имеют статус `Done`. Если нет — остановиться: эта задача не готова, взять предыдущую.
3. Если задача требует внешних действий (токен, установленные модели и т. п.) и они не выполнены — пометить задачу `Blocked`, зафиксировать причину в коммите, остановиться.

## 3. Перевести задачу в статус `Progress`

1. В `_board/plan.md` изменить поле `**Статус:** ToDo` на `**Статус:** Progress` у выбранной задачи.
2. Обновить сводную таблицу в конце `_board/plan.md` (колонка «Статус» у соответствующей строки).
3. Закоммитить изменение:
   ```bash
   git add _board/plan.md
   git commit -m "chore(plan): start task <N> — <краткое название>"
   ```
4. Создать feature-branch (если команда работает через ветки):
   ```bash
   git checkout -b task-<N>-<kebab-case-название>
   ```

> Правило: в статусе `Progress` в `_board/plan.md` одновременно находится **ровно одна** задача.

## 4. Сформулировать задачу для LLM

Требование ТЗ (`_docs/instructions.md` §1): **код пишется с помощью LLM**, не вручную.

Подготовить в чат к LLM-ассистенту (ChatGPT/Copilot/локальная модель):

1. Ссылку на раздел задачи в `_board/plan.md` (скопировать «Описание» + «Definition of Done»).
2. Релевантные выдержки из `_docs/` (архитектура компонента, сигнатуры интерфейсов, стек).
3. Уже существующие файлы, которые нужно учитывать (например, `app/config.py` при работе над `app/services/llm.py`).
4. Ограничения (`_docs/instructions.md` §10 «Что НЕЛЬЗЯ делать»).

## 5. Реализовать задачу

Итерационный цикл (`_docs/instructions.md` §1, §9):

1. **Написать/обновить тест красным.** Файлы — из `tests/`, путь зеркалит `app/` (см. `_docs/project-structure.md`). Перечень минимальных сценариев — в `_docs/testing.md` и в «Описании» задачи в `_board/plan.md`.
2. **Сгенерировать код LLM-ом** по сформулированному контексту.
3. **Довести вручную** (переименования, форматирование, мелкие багфиксы — это разрешено, полное ручное написание — нет).
4. **Прогнать тесты**: `pytest -q`. Пока красный — возвращаемся к п. 2.
5. **Проверить async-дисциплину** (`_docs/instructions.md` §4): никаких `requests`, `time.sleep`, блокирующих SDK. Только `await` на I/O.
6. **Проверить обработку ошибок** (`_docs/instructions.md` §5): все ожидаемые исключения пойманы, stacktrace — только в логи.
7. **Проверить логирование** (`_docs/instructions.md` §6): появились необходимые INFO/ERROR записи, секретов в логах нет.
8. **Проверить конфиг/секреты** (`_docs/instructions.md` §7): никаких хардкодов, всё — через `Settings`.
9. **Форматтер/линтер** (если настроен): `ruff format`, `ruff check`.

## 6. Проверить Definition of Done

1. Открыть в `_board/plan.md` раздел «Definition of Done» выбранной задачи.
2. По каждому пункту — отметить мысленно (или в `_board/progress.txt`) «выполнено / не выполнено».
3. Если хотя бы один пункт не выполнен — вернуться к шагу 5.
4. Прогнать `pytest -q` целиком (не только новые тесты) — не должно быть сломанных существующих.
5. Для задач, затрагивающих запуск бота (задачи 6+), выполнить ручную проверку в реальном Telegram: минимум `/start` и одно текстовое сообщение; для задач с командами — каждая команда из раздела.

## 7. Зафиксировать результат в git

Коммиты — атомарные, сообщения в императиве (`_docs/instructions.md` §2). Минимум:

```bash
git add <изменённые файлы>
git commit -m "feat(<scope>): <что сделано>"        # или fix / test / _docs / refactor
```

Примеры префиксов по задачам:

- Задача 3 → `feat(config): add Settings and logging` + `test(config): cover settings validation`.
- Задача 4 → `feat(llm): add OllamaClient and errors` + `test(llm): cover timeout and 404`.
- Задача 7 → `feat(handlers): add text handler with LLM call`.
- Задача 10 → `_docs(readme): add installation and commands`.

Если работали в ветке — создать PR / смёржить в `main` после проверки.

## 8. Перевести задачу в статус `Done`

1. В `_board/plan.md` у задачи сменить `**Статус:** Progress` → `**Статус:** Done`.
2. Обновить сводную таблицу в конце `_board/plan.md`.
3. Коммит:
   ```bash
   git add _board/plan.md
   git commit -m "chore(plan): complete task <N>"
   ```
4. Пуш в удалённый репозиторий: `git push`.

## 9. Обновить документацию (если нужно)

Если в ходе выполнения изменилось поведение или появились нюансы, не отражённые в `_docs/`:

1. Обновить соответствующий документ в `_docs/` (`commands.md`, `architecture.md`, `stack.md`, `requirements.md`).
2. Коммит `_docs(<scope>): ...`.

> `README.md` в корне специально полируется задачей 10. До неё обновляется только по необходимости (например, если появилась новая обязательная переменная в `.env`).

## 10. Перейти к следующей задаче

1. Открыть `_board/plan.md`, вернуться к шагу 2 настоящего документа — выбрать следующую задачу со статусом `ToDo`.
2. Если все задачи `Done` (включая задачу 10) — MVP готов: прогнать полный чек-лист из `_docs/mvp.md` §5 и отчитаться.

## Короткая шпаргалка (TL;DR)

1. Прочитать `CLAUDE.md`, `_docs/README.md`, `_board/plan.md`, `_docs/instructions.md`, документы задачи.
2. Взять верхнюю задачу `ToDo` — проверить, что зависимости `Done`.
3. Сменить статус на `Progress` в `_board/plan.md`, закоммитить (`chore(plan): start task N`).
4. Сформулировать задачу для LLM (контекст из `_docs/` + «Описание» задачи).
5. TDD: красный тест → LLM генерирует код → зелёный тест → рефакторинг.
6. Пройтись по Definition of Done построчно; прогнать весь `pytest -q`; при необходимости — ручная проверка в Telegram.
7. Закоммитить изменения с осмысленным сообщением.
8. Сменить статус на `Done` в `_board/plan.md`, закоммитить (`chore(plan): complete task N`), запушить.
9. Вернуться к шагу 2.

## Чек-лист перед переходом к следующей задаче

- [ ] Definition of Done выбранной задачи выполнен полностью.
- [ ] `pytest -q` — зелёный.
- [ ] Линтер/форматтер (если настроены) — зелёные.
- [ ] Ручная проверка в Telegram (для задач 6+) пройдена.
- [ ] Нет незакоммиченных файлов, `.env` и логи не попали в коммит.
- [ ] Статус задачи в `_board/plan.md` = `Done`, сводная таблица обновлена.
- [ ] Изменения запушены.
