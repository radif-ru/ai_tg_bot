# MVP

## 1. Цель MVP

Запускаемый Telegram-бот на Python/aiogram, который на каждое текстовое сообщение пользователя возвращает ответ локальной LLM (через Ollama), без БД и без истории диалога.

## 2. Что входит в MVP

### 2.1 Функциональность
- [x] Запуск через polling (не webhook).
- [x] Команда `/start` — приветствие + краткая инструкция.
- [x] Команда `/help` — список команд и как пользоваться ботом.
- [x] Команда `/models` — список доступных моделей.
- [x] Команда `/model <name>` — выбор активной модели для пользователя.
- [x] Команда `/prompt <text>` — установка системного промпта на процесс (или сброс к дефолту без аргумента).
- [x] Обработка произвольного текста → запрос в Ollama → ответ пользователю.
- [x] Индикатор «печатает…» (`ChatAction.TYPING`) во время генерации.
- [x] Обработка ошибок LLM (недоступность, таймаут, 4xx/5xx) с понятным сообщением пользователю.
- [x] Логирование в консоль и в файл (с ротацией).
- [x] Логирование запросов: user_id, chat_id, модель, длительность, статус.

### 2.2 Качество
- [x] Весь I/O — асинхронный (`async/await`).
- [x] Нет необработанных исключений (глобальный error handler).
- [x] Покрытие unit-тестами ключевой логики (LLM-клиент, handlers через моки).
- [x] `.env.example` в репозитории, реальный `.env` — в `.gitignore`.
- [x] `.gitignore` включает секреты, логи, виртуальное окружение, кэш.
- [x] `README.md` с инструкцией запуска и списком команд.

## 3. Что НЕ входит в MVP (явно исключено ТЗ или отложено)

- История диалога, контекст предыдущих сообщений.
- Любая БД (PostgreSQL, SQLite, Redis как persistent store).
- Облачные LLM (OpenAI, Anthropic и др.).
- Webhook-режим.
- Мультимодальный ввод (фото/аудио).
- Веб-админка.
- Деплой / Docker (можно добавить после MVP; не обязателен).

## 4. Модели по умолчанию

Из ТЗ:
- `qwen3.5:0.8b`
- `deepseek-r1:1.5b`

> Точные имена тегов могут отличаться в вашей установке Ollama (например, `qwen2.5:0.5b`, `qwen3:0.6b`). При настройке проверяйте доступность через `ollama list`. Имена моделей конфигурируются через `.env` (см. `stack.md` и `configuration` в `requirements.md`).

## 5. Критерии приёмки

MVP считается готовым, если одновременно выполнены все пункты:

1. **Запуск**: `python -m app` (или `python app/main.py`) стартует бота, polling работает, в логе появляется «Bot started».
2. **/start**: команда отвечает приветственным сообщением со списком команд.
3. **Текст → ответ**: отправка произвольного текста возвращает осмысленный ответ от выбранной LLM.
4. **Выбор модели**: `/models` показывает список, `/model <name>` переключает активную модель, следующий запрос использует новую модель.
5. **Ошибки**: при остановленной Ollama бот не падает, пользователю приходит сообщение об ошибке, в логе — запись уровня ERROR.
6. **Параллельность**: отправка 3+ сообщений подряд из разных чатов корректно обрабатывается без блокировки.
7. **Логи**: файл логов создаётся, содержит записи о входящих и о вызовах LLM с длительностью.
8. **Секреты**: в репозитории отсутствует `.env` с реальным токеном; есть `.env.example`.
9. **Тесты**: `pytest` проходит, покрыты как минимум:
   - LLM-клиент (успешный ответ, таймаут, ошибка соединения).
   - Handler `/start` и обработчик текста (с моком LLM-клиента).
10. **README.md**: содержит разделы «Установка», «Настройка», «Запуск», «Команды бота».

## 6. Определения (DoD / DoR)

- **Definition of Ready** (задача может быть взята в работу):
  - Сценарий описан в `requirements.md` или `commands.md`.
  - Понятно, какой модуль затрагивается (см. `project-structure.md`).
- **Definition of Done** (задача закрыта):
  - Код реализован и проходит линтер/форматтер.
  - Добавлены/обновлены unit-тесты; `pytest` зелёный.
  - Обновлена документация при изменении поведения.
  - Ручная проверка в реальном Telegram прошла успешно.
